# UniAuth Express Passwordless Strategy Rules

## Language

All repository-facing, Git-facing, and GitHub-facing content must be written in English only: branch names,
commit messages, PR titles and bodies, issues, labels, milestones, changelog entries,
version-controlled documentation, code comments, generated artifacts, and release notes. Local
documents that are not tracked by Git are the only exception. Russian text is forbidden in Git and
GitHub artifacts.

## Commit Messages

Use strict Conventional Commits in English only. Commit messages must describe the completed result
in past-tense/result form, not a future task or imperative instruction. Use forms such as `fixed`,
`added`, `updated`, `removed`, `hardened`, or `disabled`.

## Ownership Boundary

This repository provides passwordless-first Express strategy integration for UniAuth.

It may own:

- passwordless Express strategy composition
- public sign-in route wiring in a later change
- email OTP request and completion transport mapping in a later change
- magic-link transport mapping in a later change
- typed strategy configuration for application-owned senders

It must not own:

- auth business logic
- database lifecycle
- SMTP or provider runtime lifecycle
- OTP verification lifecycle
- magic-link verification lifecycle
- session lifecycle
- roles or permissions
- UI

## Public API

Use public `@alyldas/uniauth-core` and `@alyldas/uniauth-express` contracts only. Applications own
service construction, sender/runtime configuration, persistence, and HTTP app composition.

## Local Core Setup

Before running checks against local UniAuth packages, build upstream packages first:

```sh
cd ../uniauth-core
npm install
npm run build

cd ../uniauth-express
npm install
npm run build
```

Then return to this repository and run:

```sh
npm install
npm run check
```

## Release Metadata

`CHANGELOG.md` and `.release-please-manifest.json` are generated by Release Please. Do not edit,
hand-format, or normalize them outside Release Please release PRs unless an intentional manual
release process is being documented in the same change.

The initial scaffold seeds `CHANGELOG.md` and `.release-please-manifest.json` so Release Please has
the starting `0.1.0` package state before the first generated release PR. Future changes should let
Release Please update these files.
